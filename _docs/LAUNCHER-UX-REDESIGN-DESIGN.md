## A. 理想の体験フロー

**設計目標: 「呼び出してから視線が動く回数」を最小にする。** このアプリの1回の使用は2〜3秒。その間の視線移動と判断回数がUXのすべてであり、装飾や情報量は一切の価値を持たない。

現状のレイアウトは上から「タイトル帯 → 検索行 → タブ行 → リスト → ロゴ」の5層縦積みで、リスト先頭行は y≈75px から始まる。ウィンドウ出現時、視線はまずタイトル帯に着地し、そこから3つの境界を越えてようやくコンテンツに到達する。さらにリスト行数によってロゴ位置＝ウィンドウ高さが毎回変わるため、「n番の行がだいたい画面のこの高さ」という筋肉記憶が育たない。

新レイアウトでの1回の使用は次のようになる。

1. **出現(0秒)**: サイドボタン押下。ウィンドウはカーソルのあるモニタ中央に出る(既存挙動、変更なし)。リスト先頭行が y≈50px から始まり、**出現した瞬間に視線がリストに着地する**。上部の帯は16px+30pxの薄い2段に圧縮され「読む場所」ではなく「枠」として知覚される。
2. **分岐a — 目的の行が見えている(最頻)**: 数字キーを押して貼り付け、自動で閉じる。視線移動ゼロ。操作1打鍵。
3. **分岐b — タブが違う**: タブラベルは視線の入口である左上、ヘッダー行の先頭にある。タブをクリック(またはクリック後に数字キー)。視線移動は「左上→リスト」の1往復のみ。
4. **分岐c — 絞り込みたい**: フォーカスは出現時から検索Editにある(明示Focus、C参照)ので、**そのままタイプするだけ**でリストが絞り込まれる。検索欄はタブと同じ行にあり、「モード選択(タブ)」と「絞り込み(検索)」が1本の操作ヘッダーとして同列に見える。視線は「ヘッダー行⇄リスト」の上下1往復に収まる。
5. **終了**: 選択→貼り付け→自動クローズ(既存挙動、変更なし)。

視覚境界の数は 4→2(帯|ヘッダー、ヘッダー|リスト)。ロゴは右下隅の固定位置に常駐し、リストが3行でも10行でも**クロームの見た目が毎回同一**になる。

## B. 統合アーキテクチャ

縦積み5層を、役割の異なる4バンドに再編する。**可変高なのはコンテンツバンドだけ**という一点が構造上の核であり、他の3バンドは固定高。将来の機能追加もバンド内に収める限り再レイアウトが発生しない。

```
┌──────────────────────────────────────────────┐ y0
│ ① システム帯 h16  掴みしろ ・ バージョン ・ ⚙   │  ← 窓の管理。最も暗い色。読ませない
├──────────────────────────────────────────────┤ y16
│ ② 操作ヘッダー h30  [履歴 N][定型文 N] 🔍(検索Edit) │  ← モード選択+絞り込みを1行に統合
├──────────────────────────────────────────────┤ y46
│ ③ コンテンツ 可変   ListView(唯一の可変域)      │  ← 視線の着地点。最大面積
├──────────────────────────────────────────────┤ footerY
│ ④ ブランドフッター h32           右下にロゴ小 ▪ │  ← 固定高。リスト伸縮と完全独立
└──────────────────────────────────────────────┘
```

- **① システム帯 (h16)**: 現行の掴みしろ・歯車・バージョンを1本の連続した D4DCE8 帯に統一(現在はバージョン文字だけ帯の外に浮いている)。ドラッグ機構は既存の30msポーリングのまま。
- **② 操作ヘッダー (h30)**: Tab3のタブボタン行の右側余白(タブ2枚は左端~184pxで収まる)に、検索アイコン+Editを**重ね配置**する。Win32のタブヘッダー右側は元々何も描かれない空白であり、`UseTab()` リセット後に追加したコントロールはタブページに属さず常時表示される — ネイティブコントロールの標準的な組み合わせの範囲内で実現できる。
- **③ コンテンツ**: ListView 2枚(履歴/定型文)。オプション・幅440・列幅416・行高実測補正・ダブルバッファ・LVS_SHAREIMAGELISTS など**既存の実証済み設定は一切変えない**。
- **④ ブランドフッター (h32)**: ロゴを 32×24(73×55の縦横比を保った縮小)に小型化し、右下隅に右余白8pxで固定。アンカーは「タブコントロールの下端」ただ一つ。ListViewの座標は一切参照しない(G-1参照)。

## C. 具体機構

**① システム帯** (現行との差分: バージョン文字も帯に載せ1本化、歯車を右端へ):

## D. 偽陽性潰しの具体ロジック

検証系ではないため省略。ただし実装後の実機確認項目はE-第3歩に集約した。

## E. MVP — 実装順序

一貫した1つの設計として実装するが、リスクの独立性に沿って3歩に割る。各歩は単独で出荷・単独でロールバック可能。

**第1歩: フッター構造化(ロゴ小型化・右下固定)** — 差分約15行。
繰り返し手直しが発生してきた箇所(地雷1)を**構造ごと殺す**のが最優先。「リスト下端追従+中央揃え」を廃止し、C-④の「Tab下端アンカー+固定高バンド+ウィンドウ右端基準x」に置換する。ヘッダー改修と依存関係がなく、今日の座標のままでも入れられる。これ単体で「リスト行数によって見た目が変わる」問題が消える。

**第2歩: ヘッダー統合(検索行の廃止とタブ行への重ね配置)** — 差分約30行。
C-①②のとおり。検索行(現y18)を削除し、Tab3をy16へ、以降のy座標を28px繰り上げる。追加順の変更(Edit がTabの後になる)に伴い `LauncherSearchEdit.Focus()` を必ず入れる。

**第3歩: 実機確認(reality-checker委任)** — チェックリスト:
タブ切替10回で検索Edit/アイコンが消えたり背後に沈んだりしないか ／ 3行時と10行時でロゴが右下8px位置に不動か ／ ドラッグ移動が現行同様に効くか ／ 出現直後に即closeしないか(フォーカス監視) ／ 出現直後の即タイプで検索が効くか ／ 数字キー1〜9,0とShift付きが両タブで効くか。

## F. 捨てた案と理由

1. **検索ボックス最上部案(会議初期案)** — 却下。実使用ではタブ切替が先行する(定型文貼り付け時は必ずタブ選択から入る)ため「検索が第一操作」という前提が実態と食い違う。また検索を1行目に上げても縦積み層数は減らず、視線移動の総量が改善しない。タブと検索を**同格の操作として1行に統合**する案が、層数削減と実態適合の両方を満たす。
2. **タブごとに検索Editを持たせる案** — 却下。コントロール2重化でフィルタ状態の所有権が割れ、`LauncherFilterChanged`/`SetLauncherSearchPlaceholder` の分岐が増える。得られるものがない。
3. **タブ廃止・単一リスト+モード循環(Clibor Alt風)案** — 却下。2年かけて学習された「履歴/定型文」のメンタルモデルを破壊する割に打鍵数は減らない。白紙で構造を問い直した結果、**タブという情報設計自体は正しく、配置だけが問題**と結論。
4. **ツールバー/ReBar/オーナードローによるリッチヘッダー** — 却下。地雷4の経緯(オーナードロー系は実装コスト・回帰リスクの割に合わないとして複数回却下、ListView採用も3度目の必然があってのこと)に反する。本設計はTab3・ListView・Edit・Text・Pictureの既存5種のみで完結する。
5. **ロゴ完全撤去案** — 却下寸前まで検討。ただしマスコット/ブランドはこのアプリの人格であり、右下32×24なら視線コスト実質ゼロで残せる。コストゼロなら残す。
6. **ロゴをシステム帯(h16)に入れる案** — 却下。16pxではロゴが潰れ、掴みしろの命中率も下がる。
7. **ウィンドウ幅拡大(460→520等)案** — 却下。白紙見直しの対象にはしたが、460pxは中央ポップアップとして作業画面を隠さない実績値で、広げる積極的理由が出なかった。数値は「過去がそうだったから」ではなく「今回再検証して妥当だったから」460で確定とする。

## G. 地雷と回避策

1. **ロゴ位置の再発防止(最重要)**: 過去2回の失敗は両方「複数コントロールの実測値からロゴ位置を導出する」設計自体に起因する(Tab下端基準→隙間、修正→別コントロールに誤固定 `8e19aff`)。本設計は導出を3重に単純化して構造的に再発を断つ: (a) アンカーは**高さ調整後のTab下端の1点のみ**(LvH/LvSのGetPosとMaxを廃止)、(b) x座標は**ウィンドウ右端基準の定数**(コンテンツ非依存)、(c) ロゴは「リストに密着させる装飾」ではなく「固定高フッター帯の住人」なので、Tab下端の内部余白数pxは**仕様上の呼吸余白**であり不具合ではない。実装時、この3点をコードコメントに明記し `8e19aff` を参照させること。
2. **数値の根拠**: 本設計書の数値はすべて理由付きで再導出済み(F-7、C参照)。実装時に設計書へ「なぜその数値か」を1行ずつ残し、次回以降の見直しコストを下げる。
3. **ドラッグはポーリング続行**: システム帯の見た目は変わるが、`LauncherWatchDrag` 30msポーリング+`Gui.Move()` の機構は無改変。WM_NCLBUTTONDOWN系は実機で動かなかった実績があるため提案しない。
4. **ネイティブ縛り**: 新規コントロール種はゼロ。唯一の新技法は「Tab3ヘッダー右余白へのEdit重ね配置」だが、これもネイティブEditの通常配置(ページ非所属)である。万一実機でタブ切替時に再描画欠け(白化系)が出た場合の**フォールバックを先に決めておく**: 第1手 `LauncherTabChanged` 内で `WinRedraw`、第2手 それでもダメなら検索行をタブの上の独立行(y16, h24)に戻し全体を2行ヘッダーとする(第2歩のみロールバック、第1歩のフッターは無傷で残る)。ユーザーへの再確認は不要、この順で機械的に落とす。
5. **初期化順序**: `全コントロールAdd → Show → 実測Move(2段階中央寄せ) → WinActivate → SearchEdit.Focus() → SetTimer(CheckLauncherFocus) → 他タイマー` の順を厳守。特に今回はEditの追加順が後ろに移るため、**明示Focusの追加を忘れると「開いてもタイプできない」退行**になる。フォーカス監視タイマーが最後尾であることは現行と同一。
6. **数字キー**: 表示上のラベル(行頭の1〜9,0)は無改変。HotIfスコープ限定の常設登録方式には一切触れない。
7. **Win+V**: 本設計はキー割当に変更なし(Ctrl+Win+V・サイドボタン現状維持)。実装時にホットキー周りのコードをdiffに含めないこと。

**過剰設計への自戒**: 本設計の総差分は約45〜60行、全変更が `ShowLauncher()` 内に閉じる。新規ファイル・新規コントロール種・新規タイマー・新規メッセージフックはゼロ。「一瞬で呼び出し一瞬で使い切る」道具に必要なのは階層の削減と不動のクロームだけであり、それ以上を足さないことがこの設計の品質保証である。

# 設計: クイックペーストランチャー「ダミーデータ画面イメージ」

> 3段構え（会議ハーネス→Fable設計→実装）の成果物。COUNCIL-HOWTO.md準拠。
> 会議ハーネス素材: `tsuioku-no-kirameki.com/council/soushin-suggest-dummy-data-preview.md` / `.answers.json`
> 設計: Fable(claude-fable-5)。司令塔Claudeが実コードで裏取り済み。

対象: `index.html`の`#clipboard`セクションのみ

## 発端

ユーザーが実際のアプリ画面のスクリーンショット3枚（クイックペーストランチャーの履歴タブ・定型文タブ、「定型文の管理」ウィンドウ）を見せて、「こういうのもダミーデータとして入れれば追体験できるように」と要望した。

**重要な地雷**: スクリーンショットに写っていた定型文名（「AI会議」「kimito-linkローカルパス」等）・件数（259/261/45）・時刻は本人の実際の業務データの可能性が高く、ダミーデータへの転用は禁止。完全に架空の内容に作り替える。

## A. 現状の要約（コード裏取り済み）

- LPの`#clipboard`セクションは、テキスト説明とアイコン絵文字のみでランチャーの見た目を再現する視覚要素がゼロだった。
- LPには既にCSSモックアップの表現手法が確立済み（`.demo-window`、`.mailpreview-frame`、`.mock-file`/`.mock-tray`等）。「実物の画面をCSSで模して見せる」はLPの家風であり、会議の批判役が懸念した「デザイン一貫性を壊すリスク」は当てはまらない。
- 実アプリの表示仕様: ウィンドウ幅460px、背景`#F0F6FF`、タイトル帯`#D4DCE8`/文字`#1A3E7A`、タブに件数バッジ、検索プレースホルダーは「りんくがコピペ履歴を検索」「りんくが定型文を検索」、行頭に数字キーラベル、履歴テキストは58文字で省略、画像はサムネイル＋「画像 w×h (サイズMB)」、ゼブラ縞（奇数`#F0F6FF`/偶数`#DCE7F8`）。
- 診断ページには手を入れない（総合判定aggregateを汚さない原則、プライバシー訴求との整合を優先）。

## B. 設計判断

### 配置場所: LPの`#clipboard`のみ

診断ページは「本物の計器」であることが信頼の根拠。ダミー表示を同居させると「この画面のどこまでが本物か」という疑いを自ら持ち込むため対象外。追体験の入口はLP、実物の証明は診断ページ、と役割を分けたまま既存導線（`#transparency`→`shindan/`）で接続されている。

### ダミーデータの中身: 完全架空の事務職シナリオ

実物スクショの件数（259/261/45）は使わず、意図的に別の数字（履歴132・定型文26）にする。履歴6行（画像2・返信文の省略表示・URL・ファイルパス・架空住所）、定型文6行（一般的な事務定型文名のみ）。社名「サンプル商事」、URL`example.com`、住所「サンプル県サンプル市」など、一目で架空とわかる命名自体をサンプル明示の一部にする。

### 実装方式: 静的HTML＋CSSのみ、JSゼロ

会議で出た「JSランダム生成」案は不採用（ランダム性は閲覧者に伝わらず、静的の方が校正・保守・レビューで優位）。タブ切替のCSSハックも過剰技巧として不採用、2枚並置（デスクトップ2カラム、モバイル縦積み）にする。

### サンプル明示: 三重の明示

1. フレーム内最下部の常設帯「▲ 画面イメージ — 表示内容はすべて架空のサンプルです」
2. `figcaption`＋`aria-label`に「（サンプルデータ）」
3. 2枚の下の共通注記「※実際の画面はWindowsネイティブ表示のため、フォントや細部の見た目は多少異なります。」（誇張の芽を摘む）

## C. 実装（実際に適用したコード）

CSS: `index.html`の`.clip-use-card strong`直後に`.launcher-mock-row`系クラスを追加（背景色・タブ・行・ゼブラ・サンプル明示帯のスタイル）。`@media (max-width: 620px)`に1行追加してモバイルで1カラムに。

HTML: `.clip-launcher`の閉じタグ直後、`.clip-row`の前に、履歴タブ・定型文タブそれぞれのモック（`<figure class="launcher-mock">`）を挿入。

## D. 却下案と理由

| 案 | 却下理由 |
|---|---|
| 実スクショ画像の掲載 | 本人実データ写り込みの構造的リスク。差し替え・多言語・レティナ対応も不利 |
| 診断ページへのダミーUI統合 | 「本物の計器」への信頼を薄める。aggregate非汚染原則。プライバシー訴求と正面衝突 |
| JSランダム生成 | ランダム性は閲覧者に伝わらない。静的の方が校正・保守・レビューすべてで優位 |
| 「メール本文1」式ラベル | 訴求力ゼロ。既存の職種別カードと接続しない |
| CSSタブ切替・擬似スクロール | 過剰技巧。2枚並置の方が両タブを同時比較できる |

## E. 地雷と回避策

1. **本人データ転用の禁止**: 実スクショの定型文名・件数・時刻は一切使わない。実装済みの値は完全に架空（サンプル商事系）。
2. **誇張の禁止**: サンプル明示の帯・disclaimerは削らない。
3. **aggregate非汚染**: 診断ページには手を入れていない。
4. **CSP**: LPは静的追加のみで外部取得なし。
5. **クラス名衝突なし**: `.launcher-mock`/`.lm-*`プレフィックスで既存の`.mock-*`系と分離。
6. **LPはライトモード固定**（`color-scheme: light`）なので固定色で実装。将来診断ページへ移植する場合はダークモード分岐が必須になる点に注意。
7. **バージョン文字列**: モック内`v1.22.8`はLP内の既存モック（導入手順セクションのzipファイル名）と揃えた。AHK側の実際のバージョン（v1.23.2時点）とは異なるが、LP全体のバージョン表記更新は別の運用タスク。

- ゼブラストライプの背景色（`rgb(240,246,255)`/`rgb(220,231,248)`）が実アプリの実測色と一致
- 三重のサンプル明示（帯・disclaimer・aria-label）が全て機能
- モバイル幅（375px）で1カラムに切り替わることを確認
- 既存セクション（FAQ・価格・職種別カード）に影響がないことを確認

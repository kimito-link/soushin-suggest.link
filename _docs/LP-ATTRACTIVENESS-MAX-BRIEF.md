# Fable設計ブリーフ: LPの魅力最大化(3段構え・素材集めまで完了、Fable設計は次チャットで)

## お題
送信サジェスト(soushin-suggest.link)のLP(index.html)を、既存の作り込み(りんく・たぬ姉・こん太の世界観、SEE IT LIVE動的デモ、6シーン紹介等)を活かしたまま、「もっと魅力的に見せる」ための改善を設計してほしい。

## 決定的な事実(Explore調査で新規発見、会議のどのメンバーも知らなかった)

**LPの`final-cta`セクション(index.html 3257行目付近)に、旧キャッチコピー「右手だけで、ほとんど終わる。」が取り残されたまま残っている。** heroセクションのコピーは既に「マウスだけで、スクショもコピペも。」に変更済み(コミット`5205177`)なのに、ページ末尾の`final-cta`だけ変更が反映されておらず、**同じページ内でキャッチコピーが矛盾している**。これは今回最優先で直すべき、地味だが致命的な整合性バグ。

## その他のExplore調査結果

- LPは全3278行、hero→3人の相棒→WHO NEEDS THIS→WHAT SUGGEST MEANS→THE PROBLEM→HOW IT FEELS→SEE IT LIVE(動的デモ)→OTHER SCENES→REASONS→MOUSE MAP→ONE STROKE→CLIPBOARD&SNIPPETS(動的デモ)→SAFETY→TRANSPARENCY→BY THE NUMBERS→COMPATIBILITY→GETTING STARTED→PRICING→AFTER PURCHASE→FAQ→ALSO BY KIMITO LINK→final-cta、という非常に長い構成
- 過去に「静的CSSモックを作ったが実物感が無いと却下されてrevertされ、3段構えで動的デモに作り直した」という成功パターンがある(`LP-LAUNCHER-LIVE-DEMO-DESIGN.md`)。**この成功体験(動的デモが評価された)は今回のFable設計でも重視すべき**
- 直近実装(スクショボタン・Ctrlタップ研磨)はFAQ・mouse mapセクション・REASON 02に既に正しく反映済み。追加の文言修正は不要
- 視覚的な動的デモは「SEE IT LIVE」と「CLIPBOARD&SNIPPETS」の2箇所のみ。他の大半(3人の相棒、SAFETY、TRANSPARENCY、BY THE NUMBERS、COMPATIBILITY、PRICING、FAQ等)はテキストとキャラ吹き出しのみで、静止画像・動画は皆無
- Cloudflare Pagesでのデプロイ反映漏れ問題が過去にあり、根本原因は未確定のまま放置されている(`HANDOFF-next-session.md`参照)。LP改修を進める前に、確実に本番反映されるか要注意

## 会議ハーネスで得られた素材(4体成功)

### 統括結論(gpt-oss-120b、2ラウンド)
- 情報過多(多数のセクション)がLPの最も致命的な弱点、という批判が繰り返し提起された
- ただし「削ればいい」という単純な話ではなく、**キャラクター世界観は購入動機の一部として機能しているというユーザー調査結果**(会議内での主張、要検証)があるため、キャラクターは残す
- 最終的な統括案: 「ファーストビュー→価値提案→主要機能(1行要約)→購入ボタン→折りたたみ式FAQ」の5段階に再編成し、余計な利用シーンはアコーディオンで隠す。サイドボタン無し対応は「1クリックで切替できるビジュアルバッジ」をファーストビュー直下に配置

### 発散役(qwen3.6)のアイデア(思考のみで出力未完了、着想として使えるもの)
- 「機能説明」から「作業の物理的解放とAI連携の透明化」へのフレーム転換
- キャラクターを「機能解説者」ではなく「作業の共感パートナー」として、スクロール連動でガイド役にする
- サイドボタン無し対応を「マウス形状を選ばない」という価値として、インタラクティブな「あなたのマウスを選べ」的な見せ方にする
- 信頼性訴求として「ローカル完結・即座に元通り」を契約書風ビジュアルで見せる

### 批判役の一貫した懸念
「これ以上LPに手を入れることが本当にコンバージョンに寄与するか、情報過多で離脱を招かないか」。この懸念は最終提案で必ず正面から答えること

## 司令塔からの評価・指摘(Fableは必ず踏まえること)

1. **最優先: `final-cta`セクションの旧キャッチコピー矛盾を直す。** これはLPの魅力以前に、単純な整合性バグとして即修正すべき。設計にも明記し、実装の最初の1歩にすること
2. **会議の「情報過多が最大の弱点」という主張は、具体的なユーザー行動データ(離脱率等)の裏付けが無い、会議内の推測に過ぎない。** Fableはこれを鵜呑みにせず、「本当にセクションを削る/隠すべきか」を再検討すること。むしろExplore調査で判明した通り、**動的デモ(SEE IT LIVE、CLIPBOARD&SNIPPETS)は過去に高評価だった実績がある**ため、「情報を削る」より「まだ動的デモが無い箇所に動的要素を足す」方向のほうが、このLPの成功パターンに沿っている可能性がある
3. **大規模な構成変更(5段階への再編成、アコーディオン化)は、既に大きく作り込まれたLPへの侵襲的な変更であり、実装コストとリスクが高い。** 過剰設計にならないよう、最小の変更で最大の効果を狙う具体案を優先すること
4. **Cloudflare Pagesのデプロイ反映漏れ問題が未解決のまま**なので、LP改修を進めても本番に反映されないリスクがある。設計にはこの前提条件(実装前にデプロイ経路を確認する)を明記すること
5. これは3段構えの「手順1(会議)」までが完了した状態。**Fableでの設計(手順2)は次のチャット(コンテキストウィンドウの都合で引き継ぐ)で行う。** このブリーフ自体が次チャットへの引き継ぎ資料になるよう、自己完結的に書くこと

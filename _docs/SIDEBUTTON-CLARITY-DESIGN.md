# 設計書: LP「サイドボタン」説明の最大化

> 設計=Fable(claude-fable-5) / 素材集め=会議ハーネス(5メンバー) / 裏取り=司令塔Claude / 2026-07-14
> 3段構えワークフロー（council-fable）の手順2の産物。実装は次チャット/別モデルに引き継ぐ。

## 背景

soushin-suggest.link（AutoHotkey v2製の常駐ツール、¥980買い切り、本番稼働中）のLPで、「サイドボタン」（マウス側面ボタン、押すと全画面スクリーンショットが撮れる主要機能）の説明が実際の購入者に伝わっていなかった。X(旧Twitter)で「いまいちサイドボタンが理解できなかった」という実投稿があった。

## 実地調査で判明した事実（司令塔が裏取り済み・全て実在確認済み）

1. **サイドボタン言及箇所は5箇所、3つの独立した説明系統に分散**（デモ手順アイコン L1471-1475／mouse-mapのSVG+凡例 L2324-2378／比較表・シーンカード・価格リスト・FAQの単発言及 L1600, L2256, L2481, L2659）。統一導線がなく、1箇所を直しても他が古い表現のまま残る。

2. **SVG図解のサイドボタン位置が実物と逆側**: マウス本体はSVG座標`x=20〜180`(中心100)、サイドボタンの丸は`circle cx="163" cy="240" r="16"`（L2339）で中心より右寄り（本体右側面寄り）に描かれている。一般的な右手用マウスの実際のサイドボタン（戻る/進む）は左側面（親指側）にあるのが標準。図が逆側に描かれている可能性が高く、「理解できなかった」の一因とみられる。

3. **FAQの「戻る/進む」表記が実装と食い違う**（L2659: `サイドボタン（戻る/進む）が無いマウスの場合`）。実際のAHK実装（`dist/soushin-suggest.ahk:213`）は`XButton1::Send("#{PrintScreen}")`のみで、**XButton1（戻るボタン）専用**。XButton2（進むボタン）は無割当。事実誤認であり修正が必要。

4. **CSS/JSの名前空間衝突は無し**。`.mouse-map`・`.mouse-svg-wrap`・`.mouse-legend*`は他の自動再生パーツ(`.story-step`, `.thx-stage`)と完全独立。新規クラス名`.side-check`/`.side-btn-pulse`/`.side-hint`も衝突なし（確認済み）。

## 設計の核心（会議の対立点への回答）

会議（複数LLM）では「実写画像で位置を固定して示すべき」案と、「左手用マウス・特殊配置マウスへの誤誘導になる」という批判が対立していた。

**答え: 「位置」で定義するのをやめ、「機能」で定義する。**

本ツールの実装は`XButton1::Send("#{PrintScreen}")`のみ、つまり**OSが「戻る」として扱うボタン**だけを使う。ここが決定的で、左手用マウス・右側面配置のゲーミングマウス・メーカーユーティリティでボタンを入れ替えた環境、どれであっても「ブラウザで押すと前のページに戻るボタン」＝本ツールが反応するボタン、という等式は崩れない（AHKがフックするのはXButton1入力そのものなので、リマップ後でもブラウザの挙動と一致する）。

> **正の定義 = 「ブラウザで『戻る』が動くボタン」＋3秒セルフチェック。図解（右手用マウスの左側面・親指の位置）は「一般的な例」としての従属的な補助。**

実写固定案・透明シルエット案の対立は、どちらも「位置で教える」土俵の中の争いであり、機能定義を主に据えれば両方の批判が消える。図は「多くのマウスではここ」と親指位置を示すだけでよく、例外機種のユーザーはセルフチェックで自力解決できる。

## A. 理想の体験フロー

1. ヒーロー直下のデモ手順で「サイドボタン📸」を見る →「？」となったら手順アイコン脇の小リンク「サイドボタンとは？」で図解セクションへ飛べる。
2. mouse mapセクションに着地。マウス左わきから突き出た紫のボタンが静かに脈動しており、「親指の『戻る』ボタン」というラベルで身体感覚に接続される。
3. その直下の「うちのマウスのどれ？ 3秒チェック」枠を読む: 「ブラウザでそのボタンを押してみて、前のページに戻ったらそれです」→ 自分のマウスで即確認でき、「あ、これか」に到達。
4. 例外ユーザー（左手用・特殊配置）もセルフチェックで正しいボタンに辿り着く。「戻る」ボタン自体が無い人はFAQで「その機能だけ使えない（Win+Shift+Sで代替可）」と明確に知って安心して判断できる。

  <rect x="40" y="20" width="160" height="280" rx="80" fill="#f8fafc" stroke="#e2e8f0" stroke-width="3"/>

## D. MVP（今すぐの最小の一手）

優先順で3手。1コミットに収まる規模（HTML/CSSのみ、JS変更ゼロ）:

1. **SVGの左右修正**（C-1）— 事実と逆の図は積極的に誤解を作っているので最優先。
2. **FAQの「戻る/進む」誤記修正**（C-4）— 事実誤認の解消。

## E. 捨てた案と理由

| 案 | 却下理由 |
|---|---|
| 実写マウス画像（右手用左側面に固定） | 機種・配色の著作権/調達コスト、左手用・特殊配置ユーザーへの誤誘導、画像は今後の修正コストが高い |
| 機種形状を選べるインタラクティブパネル | ¥980買い切りLPに対して過剰設計。選択肢の網羅は不可能で、選択UI自体が新たな認知負荷 |
| 透明シルエット＋親指オーバーレイへの全面置換 | 方向性は正しいが全面作り直しは不要。既存SVGの座標修正＋親指ヒント追記で同じ効果が出る（部分採用済み） |
| 両側面対称にボタンを描く | 「どちらにもある」という嘘になり、かえって混乱を招く |
| 「サイドボタン」という語の全廃 | 検索流入・既存購入者との用語連続性を壊す。語は残し、括弧で身体感覚（親指・戻る）を接続する方式に |

## F. 地雷と回避策

1. **index.htmlは巨大ファイル（base64画像内蔵）**: 全文読み・sedでの広域抽出は禁物。Grepで行位置を特定し、Editツールで一意な文字列の局所置換のみ行うこと。`replace_all`は使わない（「サイドボタン」は文脈ごとに置換先が異なる）。
2. **SVG座標は机上では確定しない**: viewBox変更（200→210幅）と本体シフト後、必ずブラウザプレビューで「ボタンが左側面から突出して見えるか」「モバイル幅でもラベルが潰れないか」を目視確認。`x=10`の「戻る」テキストがviewBox左端で切れないか要確認（切れる場合はviewBoxを`-5 0 215 320`に調整）。
3. **アンカー飛び**: `#side-button`へのジャンプ時、固定ヘッダー（`<nav>`がstickyで存在する）があるため`scroll-margin-top`の付与を検討。
4. **リマップ環境**: メーカーユーティリティでボタン機能を入れ替えたユーザーでも「戻るが動くボタン＝反応するボタン」の等式は成立するため、文言はこのままで正しい。逆に「XButton1」「戻るボタンの物理位置」をLPに書き込むと嘘になる可能性があるので書かないこと。
5. **デモ手順のsub文言追加**はモバイルで2行に折り返す可能性がある。`.demo-step`の幅制約を確認し、崩れるなら文言を「親指の『戻る』ボタンで撮る」まで短縮する。
6. **独立コミット必須**: 他の変更と混ぜず、この設計の意図（機能定義主・図は従）をコミット本文に残す。

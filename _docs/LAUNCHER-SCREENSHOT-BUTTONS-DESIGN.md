# 設計書: サイドボタン無しユーザーへの「覚えなくていい」スクショ提供

- 設計 = Fable(claude-fable-5) / 素材集め = 会議ハーネス(3体成功) / 裏取り・統合 = 司令塔Claude
- 日付: 2026-07-23
- 3段構え(council-fableスキル)の産物。実装は次チャット/別モデルに引き継ぐ想定

## お題(なぜこの設計が必要か)

送信サジェストのLP(index.html)を精査したところ、決定的な矛盾が見つかった。

- LPのヒーローコピー・REASON 02・FAQは、いずれも「Win+Shift+Sを覚えなくていい」「ショートカットキーをひとつも覚えなくていい」ことをこのアプリの存在意義として繰り返し訴求している
- ところがFAQ(index.html 3156行)には「サイドボタンが片方しか無いマウスの場合、『進む』はWin+Shift+Sで代替できます」と書かれており、**サイドボタン無しユーザーには、このアプリが「覚えなくていい」と謳っている当のショートカットを結局覚えさせている**
- しかもWin+Shift+Sは範囲選択スクショしかできず、独自機能`CaptureMonitorAtCursorToClipboard()`(カーソルのあるモニタを範囲選択なしで即キャプチャ&履歴登録)の代替にすらなっていない

会議ハーネス(3体成功: 賛成派2体は新規ホットキー追加を提案、批判役1体は「新しいキーを覚える前提に根拠が無い」と反論)を経て、Fableが両方の立場を統合し「新規ホットキーを一切追加せず、既存の『覚えなくていい入口』であるCtrl単押しランチャーにスクショ機能を統合する」という第三の道を設計した。

## 結論

**新規ホットキーは1つも追加しない。** ランチャーのフッターに「画面を撮る」「範囲を撮る」の2ボタンを追加し、既存のスクショ処理関数をそのまま呼び出す。

- サイドボタン有り: 1アクション(ボタン1押し)で撮影(変更なし)
- サイドボタン無し: **Ctrlポン→ランチャーのボタンをクリック**の2アクションで、同一品質のスクショ機能に到達

これにより「ショートカットキーをひとつも覚えなくていい」がサイドボタン無しユーザーにも真になる。Ctrl単押しは「組み合わせを覚える」に当たらない、という扱いはLP自身が既にCtrl単押しランチャーの文脈で採用済み。

## 実装内容

### 1. XButton2ハンドラの中身を関数抽出(dist/soushin-suggest.ahk 855-867行付近)

現状ベタ書きの短押し/長押し処理を、`DoFullScreenShot()`/`DoRegionSnip()`として抽出する。`LastUserCopyTick`の記録とフォールバックを**丸ごと**移すこと(検疫フィルタを通すための記録を片方だけ更新する二重管理を防ぐ)。

```ahk
DoFullScreenShot() {
    global LastUserCopyTick
    LastUserCopyTick := A_TickCount
    if !CaptureMonitorAtCursorToClipboard()
        Send("#{PrintScreen}")
}
DoRegionSnip() {
    global LastUserCopyTick
    LastUserCopyTick := A_TickCount
    Send("#+s")
}
```

XButton2ハンドラはこの2関数を呼ぶだけに置き換える。

### 2. ランチャーのフッターにボタン2つ追加(1784-1791行付近)

現在のフッターはロゴのみ(Tab3下端をアンカーにした1点導出、_docs/LAUNCHER-UX-REDESIGN-DESIGN.md参照)。ロゴの左右にButtonを配置する:

```
[📷 画面を撮る]   (ロゴ)   [✂ 範囲を撮る]
```

- クリック時のフロー: `CloseLauncher()` → `Sleep(150)` → `DoFullScreenShot()`または`DoRegionSnip()`
- 撮影前に必ずランチャーを閉じる。閉じずに撮るとランチャー自身が写り込む上、+AlwaysOnTop同士のZ-order競合でListViewが白く残る既知の不具合(`CaptureMonitorAtCursorToClipboard`内コメント参照)を誘発する
- `LauncherTab.UseTab()`でタブ非所属に戻してから追加する(ロゴと同じ扱いにしないと片方のタブでしか表示されない)
- 透明スペーサ(現1791行)の高さにボタン行分を含める。含めないとウィンドウ全体の高さ計算が狂い見切れる

## 捨てた案と理由

1. **Ctrl+Alt+S等の新規ホットキー(会議賛成派2体の案)** — 「ショートカットキーをひとつも覚えなくていい」という商品の存在意義と正面矛盾。他アプリ(保存系機能)との衝突リスクもある
2. **デフォルト無効のオプションホットキー(会議批判役の案)** — 発見されず存在しないのと同じ。設定画面にホットキー概念を持ち込むことになり、有効化の先が結局「覚える」作業になる
3. **Ctrlダブルタップ** — 既存のCtrl単押しランチャーと判定が競合する
4. **Ctrl長押し** — Ctrl+クリックやCtrl押しっぱなしのドラッグは日常操作であり誤爆必至
5. **ミドルクリックへの割り当て** — `MButton::ActivateGitBash()`が既に使用中
6. **PrintScreenキーの活用案内** — Fn同時押しが必要なノートPCが多く普遍性が無い
7. **FAQ文言の修正のみ(機能は現状維持)** — 矛盾の根本(機能欠落)が残る
8. **トレイメニューにスクショ項目を追加** — トレイまでの移動が長く発見性が低い。ランチャー案の完全下位互換

## 実装時の地雷

1. **Ctrl+Win+Vの前科に近寄らない** — 2026-07-18にWindows標準機能と衝突して撤回済み(コード内コメント参照)。本設計はキー追加ゼロが骨子、実装中に「ついでにホットキーも」を足さないこと
2. **撮影前に必ず`CloseLauncher()`→`Sleep(150)`を挟む** — 順序を守らないとランチャーが写り込む、またはZ-order競合で白化する
3. **XButton2ハンドラの中身は丸ごと関数抽出する** — `LastUserCopyTick`の記録を片方だけ更新すると検疫フィルタに弾かれ「撮れたのに履歴に載らない」バグになる
4. **フッターのY座標はTab3下端の1点アンカーを維持する** — 過去に「リストの実測下端からロゴ位置を導出→間違ったコントロールに固定→再修正」を2回繰り返した記録がある(コミット8e19aff)。ボタン行を足す際も導出元を増やさない
5. **ボタンは`LauncherTab.UseTab()`リセット後に追加する** — タブ所属のまま追加すると片方のタブでしか表示されない
6. **WS_EX_COMPOSITEDを再提案しない** — チラつき対策として過去に導入し、常時白化を招いて撤回済み(コミットe279f14)

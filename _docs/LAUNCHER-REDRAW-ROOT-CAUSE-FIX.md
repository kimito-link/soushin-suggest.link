# ランチャー再描画バグ 根本原因調査と対策(記録)

調査=司令塔Claude(Explore/Planエージェント活用) / 2026-07-18 / 3段構え(council-fable)は使わず、
直接調査・設計・実装したケース。次回同種の「一部だけ消える」系不具合が出た場合、対症療法へ
逆戻りしないための記録として残す。

## 経緯

サイドボタン(XButton2)でスクリーンショットを撮ると、ランチャーGUIの一部が再描画されず消える不具合が
3回連続で再発した。

1. 最初: サイドボタンでスクショを撮るとランチャー内のリスト(ListView)が真っ白になる
   → `FlashScreenRect`(暗転演出用のAlwaysOnTopウィンドウ)とランチャーのZ-order競合が原因と特定し、
   「ランチャー表示中は演出をスキップ」で対処 → 後にユーザーから「演出が無いと撮れたか分からない」と
   指摘され、演出を常時表示に戻す代わりに`PushClipImage`内で`SetTimer(RefreshLauncherHistory, -1)`
   によるリスト強制再描画を追加して解決したはずだった。
2. 次: 定型文タブでも同じ白化が再現 → `RedrawActiveLauncherSnippetsTab`という同種の強制再描画を
   追加して解決したはずだった。
3. 直近: ランチャー全体のUX再設計(検索ボックスをタブ行に重ね配置)を行った後、再度「サイドボタンで
   スクショを撮るとヘッダー(タブ・検索欄)が消える」症状が再発。ウィンドウ全体をInvalidateRectする
   `RedrawLauncherWindow`で対処しようとしたら「今度はリストが真っ白になる」新症状が発生。次に
   `RedrawLauncherHeader`(Tab3コントロールだけをInvalidateRect)に変更したらヘッダーは直ったが、
   今度は「検索ボックスの下端の枠線が消える」症状が発生、RDW_FRAMEを追加しても直らなかった。

個別コントロールを狙った対症療法を3回試したが、直すたびに「別の箇所が消える」形で症状が
移動し続けた。

## 根本原因

**ListView(履歴タブ・定型文タブ)だけが`+LV0x10000`(LVS_EX_DOUBLEBUFFER)でダブルバッファ防御
されており、Tab3(タブコントロール)・検索Editにはこの防御が一切無かった。**

これまで「白化バグ」が履歴タブ→定型文タブと再発したのは、対象がいずれもListViewで、都度
ダブルバッファ化すれば解決していたため。直近のUX再設計(検索欄をTab3の上に重ね配置)で、初めて
「元々無防備だった非ListView領域」が可視の被害範囲に入っただけであり、これは新しいバグではなく
「対症療法のたびに対象コントロールを変えて再発するモグラ叩き」だった。個別コントロール単位の
対症療法を続ける限り、将来追加するどんなコントロールでも同じ現象が起きる構造的欠陥だった。

副次要因: `FlashScreenRect`が生成する`+AlwaysOnTop`の暗転ウィンドウ自体が、Z-order競合の
直接の引き金になっている。RDW_FRAMEを追加しても直らなかった事実は、問題が「再描画命令の種類」
ではなく「命令を送るタイミングとZ-order状態そのもの」にあることを示唆していた。

## 検討した対応方針

**追記(2026-07-18、コミット`e279f14`): 下表でA(採用)としたWS_EX_COMPOSITEDは、実機で「スクショ操作と無関係にランチャーを開くたび常に真っ白になる」という別の重大な副作用を引き起こすことが判明し、撤回した。** 撤去後は個別コントロールへの対症療法(`RedrawActiveLauncherSnippetsTab`・`RedrawLauncherHeader`等)に戻っており、本ドキュメントが「根本解決」と呼んでいたスクショ時白化バグは**未解決のまま**である。詳細経緯は`dist/soushin-suggest.ahk`のL1437-1442・L1815-1819のコメント参照。**WS_EX_COMPOSITEDは再提案しないこと。**

| 案 | 内容 | 判定 |
|---|---|---|
| A(採用→撤回) | ランチャーウィンドウ全体に`WS_EX_COMPOSITED`(0x02000000)を付与し、DWM合成による構造的ダブルバッファ対応にする | **撤回済み(2026-07-18)**。この環境で常時白化という別の重大な副作用を実機で確認し撤回。当初は個別対症療法をやめる狙いで採用したが、対症療法に逆戻りしている |
| B | `FlashScreenRect`のAlwaysOnTopウィンドウ生成自体を避ける | 却下。暗転演出を`+AlwaysOnTop`にしないとランチャー表示中は演出が隠れ、「暗転演出は撮れた実感として必須」というユーザー要件に抵触するリスクが高い |
| オーナードロー等のカスタム描画 | Tab3・Editを自前描画に置き換える | 却下。過去に複数回「実装コスト・回帰リスクの割に合わない」として却下されている実績があり(`_docs/LAUNCHER-SNIPPETS-LISTVIEW-DESIGN.md`)、今回も同じ判断 |

## 実装内容

対象: `dist/soushin-suggest.ahk`

1. `ShowLauncher()`内、`LauncherGui := Gui(...)`の直後にWS_EX_COMPOSITEDを付与:
   ```ahk
   GWL_EXSTYLE := -20, WS_EX_COMPOSITED := 0x02000000
   exStyle := DllCall("GetWindowLong", "Ptr", LauncherGui.Hwnd, "Int", GWL_EXSTYLE, "Int")
   DllCall("SetWindowLong", "Ptr", LauncherGui.Hwnd, "Int", GWL_EXSTYLE, "Int", exStyle | WS_EX_COMPOSITED)
   ```
2. `PushClipImage()`末尾の`RedrawLauncherHeader`を、個別コントロール(Tab3・検索Edit)への
   ピンポイント対症療法から、ウィンドウ全体への単純な`InvalidateRect`に統合。WS_EX_COMPOSITED
   適用後はこの呼び出し自体不要になっている可能性が高いが、保険として残し、将来コントロールが
   増えても本関数を書き換える必要がない形にした。

変更しなかったもの:
- ListViewの`+LV0x10000`(LVS_EX_DOUBLEBUFFER)はそのまま残した(WS_EX_COMPOSITEDと機構が
  重複するが実害のない冗長性であり、削除するリグレッションリスクを取る理由がないため)。
- 暗転演出(`FlashScreenRect`)自体は削除・スキップ化していない。
- `Move()`をListBox等に事後実行する既知の別地雷には触れていない。

## 教訓(次回同種の不具合が出た場合に踏まえること)

「一部のコントロールだけ再描画されず消える」系の不具合が出た場合、個別コントロールへの
InvalidateRect/RedrawWindowでの対症療法を安易に繰り返さない。まず「なぜそのコントロールだけ
無防備なのか」を疑い、既にダブルバッファ対応済みの他コントロール(ListViewの`+LV0x10000`等)との
非対称性がないか確認すること。同じパターンが3回起きた時点で、個別対応ではなくウィンドウ全体の
構造的対策(`WS_EX_COMPOSITED`)を検討する。

# 設計: クイックペーストランチャー動的デモ（動画みたいなアニメーション）

> 3段構え（会議ハーネス→Fable設計→実装）の成果物。COUNCIL-HOWTO.md準拠。
> 会議ハーネス素材: `tsuioku-no-kirameki.com/council/soushin-suggest-launcher-live-demo.md` / `.answers.json`
> 設計: Fable(claude-fable-5)。司令塔Claudeが実コードで裏取り済み。

対象: `index.html`の`#clipboard`セクションのみ

## 発端

1. ユーザーが実際のアプリ画面のスクリーンショット3枚を見せて「ダミーデータで追体験できるように」と要望
2. 最初に静的CSSモック（HTML/CSSで再現した2枚の画像的表示）を実装したが、ユーザーの反応は「これじゃない、動画みたいな動的なものが欲しい」だった。静的モックは撤回済み（`git revert`）
3. 調査の結果、LPには既に本格的なJSアニメーションデモ「SEE IT LIVE」（`.thx-stage`）があることが判明。今回はこれと同じ仕組み・トーンで新シーンを作る

## A. 現状の要約（コード裏取り済み）

- `.thx-stage`（1877行〜）: フェーズ関数のリレー方式（`seg`/`at`ヘルパー）、CSS transition駆動のカーソル移動・長押しリング・トースト・HUD・ペイオフ・リプレイボタン、`prefers-reduced-motion`の二重対応、rectベース自動再生（IntersectionObserverではなくスクロール監視）。
- **重要な技術的発見**: 既存の`setCursorAt(xPct, yPct)`は`transform: translate(x%, y%)`を使うが、CSSの`translate()`のパーセンテージは**要素自身のサイズ基準**（MDN仕様どおり）。つまりカーソル要素（26px四方）の26%移動にしかならず、ポップアップ内の特定の行を正確に狙う用途には使えない。新シーンでは`left`/`top`（親要素基準の絶対配置）方式に変更する必要がある。
- `#clipboard`セクション（2494行〜）の`.clip-launcher`説明文の直後が、動的デモの自然な挿入位置。

## B. 設計判断

### 操作シーケンス: 「2幕構成」で速さと賢さの両方を見せる

- **幕1（約5.2秒）**: 長押し→ポップアップ→3番目の項目をクリック→貼り付け（基本操作）
- **幕2（約4秒）**: 同じランチャーを再度開くと、幕1で使った項目が先頭に昇格している。数字キー「1」で即貼り付け

理由: 「速さ」単独は既存の「SEE IT LIVE」と訴求が重複する。`#clipboard`セクション固有の価値は「履歴が学習して手に馴染む」ことで、直近実装した「ペーストで先頭昇格」機能（`4090837`）はまさにその差別化点。幕2で数字キー、幕1でマウスクリックと役割分担することで、説明文の「クリックか、数字キー」の両方をカバーする。

### アーキテクチャ: 完全独立IIFE。コンポーザーは却下

新シーンは独立した`<div class="thx-stage qp-stage">`＋独立IIFEとし、`played`フラグ・`timers`配列・自動再生リスナーはすべてIIFE内ローカル。既存の見た目が同一の部品（ステージ枠・カーソル・リング・トースト・HUD・ペイオフ・リプレイ）はCSSクラスとして共有し、ランチャー固有の新規部品は`.qp-`プレフィックスで新設。既存JSの`document.querySelector('.thx-stage')`は文書順で最初の1個を取得するため、新ステージは既存より後方に配置すれば競合しない（既存コードは1文字も変更しない）。

「モジュラーシーンコンポーザー」（会議の発散案）は却下: 動く既存コードのリファクタが必須で「壊さない」原則に反する。シーンが2つでは抽象化の損益分岐点に達していない。3つ目の動的シーンが出た時点で再検討する。

### 尺とループ: 1回自動再生→静的結果→リプレイボタン。無限ループは却下

既存「SEE IT LIVE」と同じ方式（前例踏襲）。無限ループは離脱要因・CPU浪費になるため却下。既存にある「4秒後の無条件自動再生フォールバック」は新シーンには入れない（`#clipboard`はページ後半にあり、画面外で再生が終わる事故になるため）。

### ポップアップ出現演出: シンプルなfade+scale

`opacity:0; transform: scale(0.94) translateY(6px)` → `.is-open`で`opacity:1; scale(1) translateY(0)`、軽いオーバーシュートのcubic-bezierで「ポン」感を出す。「プログレッシブリビール」「モーフ」（会議の発散案）は却下: 「長押し→開く」の因果が濁る。モーフはCSS transitionだけでは実装できず既存の設計規約から逸脱する。

## C. 実装（実際に適用したコード概要）

- **ダミーデータ**: 完全架空の5項目（会議URL・お礼メッセージ・定例変更連絡・フォルダパス・電話番号サンプル）。ユーザーの実スクショ由来のデータは一切使用しない。
- **HTML**: `.clip-launcher`説明文の直後に`.qp-stage`を挿入。架空チャット画面（`chat.example.com`）、ポップアップ、カーソル、トースト、ペイオフ、リプレイボタン、注記文を含む。
- **CSS**: `.qp-`プレフィックスで新規スタイルを追加。カーソルは`left`/`top`のtransitionに変更（既存の`transform: translate(%)`の限界を回避）。
- **JS**: 独立IIFE。`renderList`/`cursorTo`/`startWatch`/`flashMouse`/`resetVisuals`/`play`の各関数。`played`/`timers`はローカル変数。`prefers-reduced-motion`時は`showStaticResult()`で最終結果を静的表示。

## D. 実装順序

1. CSS＋HTML骨格（JSなしで見た目・レスポンシブ確認）
2. シーンJS（`rowY()`のY%座標を実レイアウトに合わせて微調整）
3. reduced-motion確認
4. 自動再生とリプレイの動作確認（既存「SEE IT LIVE」との相互干渉がないことも確認）
5. 文言最終チェック（ダミーデータ・誇張表現の目視確認）

## E. MVP

2幕＋ペイオフ＋静的フォールバック＋リプレイボタンで完結。定型文タブ切替演出・ホバープレビュー・キャラクターのセリフはMVP後の後続タスク。

## F. 却下案と理由

| 案 | 理由 |
|---|---|
| 速さ訴求のみ | 既存SEE IT LIVEと訴求重複。昇格という固有価値が出ない |
| モジュラーシーンコンポーザー | 既存コードのリファクタが必須。シーン2つでは抽象化の損益分岐点未満 |
| 無限ループ再生 | 離脱要因・CPU浪費・既存ステージとの作法不一致 |
| プログレッシブリビール／モーフ | 因果が濁る。モーフはCSS transitionで実装不能 |
| 既存`setCursorAt`(translate%)の流用 | 要素自身サイズ基準で行の狙い撃ちに使えない（コード裏取りで確認） |
| 4秒後の無条件自動再生 | ページ後半セクションでは画面外再生事故になる |
| 診断ページへのデモ追加 | 総合判定を汚さない原則（前回設計踏襲） |

## G. 地雷と回避策

1. **reduced-motion二重対応**: グローバルCSSは自動で効く。JS側は`matchMedia`判定で`showStaticResult()`へ分岐。
2. **スコープ分離**: `played`/`timers`/リスナーは新IIFE内ローカル。新ステージは既存より必ず後方に配置（HTMLコメントで明文化）。
3. **モバイル**: ポップアップは`width: min(320px, 82%)`、620px以下で`width: 90%`。カーソルは親%基準のleft/topでリサイズ耐性あり。リスト項目に`white-space: nowrap; text-overflow: ellipsis`で行高固定。
4. **ダミーデータ**: 全項目を架空に新作。実スクショ由来のデータは不使用。
5. **誇張禁止**: ペイオフは操作の対比のみ。秒数の水増しマッピングは使わず実時間表示。
6. **プライバシー誤読防止**: アプリバーに「デモ用の架空画面」、ペイオフとステージ下部に注記。
7. **診断ページ**: 一切触らない。変更ファイルは`index.html`のみ。

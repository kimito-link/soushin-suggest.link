## A. 理想の体験フロー

**変わること**: 「アプリでコピー → ページでCtrl+V」の2手が消える。

1. **初回**: トレイメニュー「診断ページで見る」を選ぶと、確認ダイアログが1回だけ出る。
   > 「診断カウンター(動作回数と設定値のみ。クリップボードの中身・履歴・個人情報は含みません)を soushin-suggest.link に送信し、ブラウザで表示します。サーバー上のデータは6時間で自動削除されます。送信してよいですか?」
   [送信して開く] / [キャンセル]
2. **同意後**: アプリが診断JSONを送信し、既定ブラウザで `https://soushin-suggest.link/shindan/#t=<トークン>` が開く。ページを開いた瞬間に最新状態が色分けカードで表示されている。貼り付け操作は存在しない。
3. **自動更新(第2段階)**: 設定で「自動送信」をONにすると、ページを開きっぱなしにするだけで数分おきに最新値へ更新される。ONにする操作のたびに上記と同種の確認が入る。
4. **従来の手動貼り付けは残す**: オフライン環境・送信を望まないユーザーのため、貼り付け欄はページから消さない。自動化は「追加の入口」であり、置き換えではない。

ユーザーから見た約束: 送るのはカウンターだけ、送るのは自分が許可したときだけ、サーバーには6時間しか残らない、ページのURLを知らない他人は自分のデータに到達できない(トークンは128bitランダム)。

## B. 統合アーキテクチャ

```
[アプリ AHK v2]                | [Cloudflare Pages]                  | [ブラウザ]
                               |                                     |
DiagPush(新規 ~100行)          | functions/api/diag/report.ts (新規) |  shindan/index.html (改修)
 ・起動時に128bitトークン生成   |   POST: 検証→KV.put(TTL 6h)        |   URLフラグメント#t=からトークン取得
   (メモリのみ・非永続)        |                                     |   GET /api/diag/latest?t= を
 ・BuildDiagText()を再利用     | functions/api/diag/latest.ts (新規) |   表示中のみポーリング
 ・WinHttpRequestでPOST        |   GET: KV.get→200/404              |   既存の描画ロジックはそのまま再利用
 ・失敗しても本体を止めない     |                                     |   手動貼り付け欄は存続
                               | KVネームスペース DIAG_KV (新設)     |
                               | (wrangler.toml or ダッシュボード)   |
```

役割分担の要点:
- **アプリ**: 「いつ送るか」の決定権とトークンの生成権を持つ。トークンはプロセス起動ごとに新規生成しメモリのみに保持 — **非永続の原則をトークンにも適用**する。再起動すれば旧データは孤児化しTTLで消える。
- **サーバー**: 素通しの一時置き場。`diag:<token>` キーに最新1件を上書き保存するだけ。集計しない・ログに残さない・トークンを跨いだ関連付けをしない。
- **ページ**: トークンをURL**フラグメント**(`#t=`)で受け取る。フラグメントはHTTPリクエストに含まれないため、Cloudflareのアクセスログにすらトークンが残らない。既存の解析・カード描画コードは入力源が「貼り付け」から「fetch結果」に増えるだけで共通。

## C. 具体機構

### C-1. 匿名トークン
- 生成: 起動時に `BCryptGenRandom`(DllCall)で16バイト → 32桁hex。AHKの`Random()`はCSPRNGでないため使わない。
- 保存: グローバル変数のみ。settings.ini・レジストリ・ファイルに書かない(書けば擬似デバイスIDになるため**禁止**)。
- 送信データにHWID・MAC・ユーザー名・ホスト名・IPの明示送信は一切含めない。ペイロードは既存`BuildDiagText()`の出力そのまま(カウンター+設定値のみ、という既存の保証を流用)。

### C-2. エンドポイント(完全隔離パス `/api/diag/*`)
`_routes.json` は既に `"include": ["/api/*"]` なので**変更不要**。`functions/api/diag/` に新規ファイルを置くだけで、`functions/api/stripe-webhook.ts` には1文字も触らない。

- **POST `/api/diag/report`**
  - Body: `{"token":"<32hex>","diag":{...BuildDiagTextの出力...}}`
  - 検証(fail-closed、1つでも落ちたら400で破棄): `Content-Type: application/json` / body ≤ 8KB / `token` が `^[0-9a-f]{32}$` / `diag.app === "soushin-suggest"`
  - 処理: `env.DIAG_KV.put("diag:"+token, JSON.stringify({receivedAt: Date.now(), diag}), {expirationTtl: 21600})` → 204
- **GET `/api/diag/latest?t=<32hex>`**
  - トークン形式検証 → `KV.get` → あれば200(JSON)、なければ404。`Cache-Control: no-store` 必須。
- **保持期間**: TTL 21600秒(6時間)。会議の「24時間以内」より狭く取る。削除処理の実装は不要 — KVのTTLに全委任(消し忘れが構造的に起きない)。
- **KVバインディング**: `wrangler.toml` を新規作成し `[[kv_namespaces]] binding = "DIAG_KV"` を定義(リポジトリに設定の正本を残す)。

### C-3. AHK側HTTP実装(新規モジュール、fail-closed)
```ahk
DiagPush() {
    try {
        req := ComObject("WinHttp.WinHttpRequest.5.1")
        req.SetTimeouts(2000, 2000, 2000, 3000)   ; resolve/connect/send/receive
        req.Open("POST", DIAG_ENDPOINT, false)
        req.SetRequestHeader("Content-Type", "application/json")
        req.Send(payload)
        DiagCount(req.Status = 204 ? "diagPush.ok" : "diagPush.rejected")
    } catch {
        DiagCount("diagPush.fail")               ; 失敗も計器に載せる(嘘をつかない診断)
    }
}
```
- URL(`DIAG_ENDPOINT`)はコード内定数。**設定ファイルから読まない**(sites.ini/settings.ini改ざんで任意サーバーへ送信させられる注入経路を作らない)。
- 同期呼び出しだがタイムアウト合計は最悪~9秒。これを踏まないため: 送信は必ず`SetTimer(..., -1)`の負値ワンショットタイマー経由で呼び、ホットキー・クリップボード監視のコールパスから直接呼ばない。**2回連続失敗したら当該セッションの自動送信を停止**(トレイに「自動送信を一時停止しました」通知、手動送信は引き続き可)。リトライループは作らない。
- 送信頻度: 手動送信=操作時のみ。自動送信ON時=**5分間隔**(KV無料枠1000 writes/日に対し288/日で余裕。60秒間隔は枠超過するため不可)。「診断ページで見る」実行時は間隔に関係なく即時送信してから開く(開いた瞬間の鮮度を保証)。

### C-4. 設定キー(既存SettingDefsパターンに乗せる)
```
"diag.autoSend",  {section: "diag", default: "off", apply: ApplyDiagAutoSend}
"diag.consented", {section: "diag", default: "no",  apply: ...}
```
- **デフォルトOFFの二重担保**(設定ファイル破損に依存しない):
  1. 送信可否は `値 = "on" との完全一致` でのみtrue。キー欠落・空文字・文字化け・ファイル消失 → すべてOFF側に落ちる(許可リスト方式)。
  2. さらに `diag.consented = "yes"` が同時に成立しない限り自動送信タイマーは起動しない。consentedは確認ダイアログの[同意]でのみ書かれる。ファイル破損でconsentedが消えた場合 → 自動送信は止まり、次回ON操作時にダイアログが再表示される。**壊れた結果が「もう一度確認が出る」方向にしか倒れない**。
  3. 起動時に自動でダイアログを出すことはしない(起動のたびの確認疲れ→反射的同意、を避ける)。確認はユーザーがON操作した文脈でのみ。

### C-5. ページ側
- CSP変更: `connect-src 'none'` → `connect-src 'self'`(自オリジンのみ。外部ドメインは引き続き遮断)。
- `#t=` があれば: 初回fetch → 描画。`document.visibilityState === "visible"` の間だけ30秒間隔でポーリング、非表示で停止。404なら「アプリからまだ届いていません(トレイの『診断ページで見る』を実行してください)」を表示。
- `#t=` がなければ: 現行どおり貼り付けUIのみ。
- C-6節プライバシー文言の全面書き換え(現「送信するコード自体が存在しません」は虚偽になるため削除):
  > 「このページはあなたが『診断ページで見る』を実行したときだけ、動作カウンターと設定値(クリップボードの中身・履歴・個人情報は含みません)を soushin-suggest.link のサーバーに送って表示します。データはランダムな一時トークンにのみ紐づき、6時間で自動削除されます。送信しない使い方(手動貼り付け)もそのまま使えます。」

## E. MVP — 中間案から始める

**第1歩(これだけで「貼り付け廃止」の体験は完成する)**: 会議の中間案そのもの = **明示ボタン送信のみ、バックグラウンド常時送信なし**。

1. `wrangler.toml` + DIAG_KV作成、`functions/api/diag/report.ts` / `latest.ts` 実装(curlで動作確認可能)
2. `shindan/index.html`: CSP変更・`#t=`読み取り・fetch描画・プライバシー文言書き換え
3. アプリ: トークン生成 + `DiagPush()` + トレイ項目「診断ページで見る」(初回ダイアログ→送信→ブラウザ起動) + `diag.consented` キー

この段階では `diag.autoSend` も5分タイマーも**実装しない**。送信は毎回ユーザーの操作起点なので同意設計が最も単純で、失敗時の影響もその操作1回に閉じる。

**第2歩(第1歩の運用で問題なければ)**: `diag.autoSend` + 5分タイマー + 連続失敗時自動停止 + ページ側ポーリング。

**やらないこと(過剰設計の戒め)**: 複数デバイス一覧・履歴グラフ・時系列蓄積・D1/Durable Objects・WebSocket/SSEプッシュ・リトライキュー・送信データの暗号化層追加(TLSで足りる)。すべて「最新1件が見えれば診断は成立する」という目的を超える。

## F. 捨てた案と理由

| 案 | 捨てた理由 |
|---|---|
| **実デバイスID(HWID/MAC)を識別子に使う** | 会議の最重要警告。匿名性が崩れ、サーバー側でユーザー個体の追跡が構造的に可能になる。「オフライン完結・非永続」を覆した今、残された信頼の軸は匿名性と短期保持であり、これを守れないなら転換自体をやめるべきレベルの逸脱。 |
| **トークンをsettings.iniに永続化** | 再インストールしても同じIDが送られ続け、擬似デバイスIDに変質する。プロセス起動ごと再生成なら「アプリ側は非永続」の原則がトークンにも一貫する。トークンが変わる不便は「診断ページで見る」がURLごと開き直すので実害なし。 |
| **長期保持(数日〜)・時系列蓄積** | 「アプリは非永続」と謳いながらサーバーでこっそり永続化する矛盾(地雷4)そのもの。診断の目的は「今の状態」の共有であり、最新1件+6時間TTLで足りる。 |
| **D1 / Durable Objects** | スキーマ・マイグレーション・削除バッチの管理が発生。KVは put時のexpirationTtlだけで「自動削除」が仕様として保証され、削除の実装も消し忘れも存在しない。要件に対して唯一「何もしないで済む」選択肢。 |
| **ローカルHTTPサーバー(127.0.0.1)をアプリに立て、ページがlocalhostへfetch** | 外部送信ゼロで自動化できる魅力はあるが、https配信ページからの混在コンテンツ制限・CSP・ファイアウォール警告・AHKでのHTTPサーバー実装コストで現実性が低い。ただし「外部送信ゼロ」の価値は大きいため、将来ネイティブ再実装等があれば再検討に値する(記録として残す)。 |
| **貼り付け自動化(アプリがブラウザを操作してCtrl+V)** | UI自動化は壊れやすく、フォーカス奪取の副作用が本体の信頼性を毀損。 |

## G. 地雷と回避策(地雷マップ7項目に対応)

1. **製品方針の転換の明記** — 本書冒頭に転換日(2026-07-18)・ユーザー明示同意・旧却下理由を記載済み。`_docs/SHINDAN-VIEWER-DESIGN.md` F節に「2026-07-18の合意により本判断は覆された。現行正本は本書」と追記すること。黙って上書きしない。
2. **保存インフラ新設** — `wrangler.toml` をリポジトリに新規作成しKVバインディングを正本化。ダッシュボード手作業だけに依存させない。デプロイ前にプレビュー環境でKVバインディング名の不一致(典型的な500の原因)を確認。
3. **AHK通信コードは完全新規** — C-3のとおり: タイムアウト厳格(合計~9秒上限)・ワンショットタイマー経由・try/catch必須・連続失敗で自動停止・リトライなし。送信失敗がクリップボード監視やホットキー応答を1msでも遅らせる実装(監視コールバック内からの直接送信)を禁止。失敗自体を`ClipDiag`カウンターに記録し、診断が自分の不調も申告する。
4. **非永続原則との矛盾** — アプリ非永続(トークン含む)+サーバーTTL 6時間+最新1件上書きの3点セットで「どこにも蓄積されない」を維持。サーバー側でアクセスログ以外の独自ログ・分析を追加しないことを設計上の禁止事項として明記。
5. **CSPと文言** — `connect-src 'self'` への変更と、C-5の新文言への全面書き換えを**同一コミット**で行う(コードと説明が食い違う瞬間を作らない)。「送信するコード自体が存在しません」の残骸検索(`grep "存在しません"`)をリリース前チェックに入れる。
6. **G-9制約の解除とStripe隔離** — 本書をもって「`functions/api/`に触れない」制約を解除する。ただし実装は `functions/api/diag/` 配下の**新規ファイル追加のみ**とし、`stripe-webhook.ts` と `_routes.json` は差分ゼロ(routesは既に`/api/*`をincludeしており変更不要)。PRレビューで両ファイルの差分がないことを機械的に確認。
7. **設定パターンの流用と破損耐性** — `SettingDefs` に素直に乗せる(C-4)。担保の核心は「**送信条件は 'on' との完全一致 かつ consented='yes' の二重AND、それ以外の全状態はOFF**」という許可リスト構造にあり、iniの壊れ方がどうであれ倒れる先は常に「送信しない+確認ダイアログ再表示」。この不変条件をコードコメントとテスト(settings.ini削除・文字化け・空ファイルの3ケースで送信タイマーが起動しないこと)で固定する。
